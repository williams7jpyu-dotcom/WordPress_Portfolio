初めてのmergeコンフリクト

5人のチーム実習。自分のブランチをマージしようとしたら、赤い文字でCONFLICTと出た。最初は事故かと思った。

個人で課題をやっているときは、Gitは「保存ボタン」くらいの感覚だった。addしてcommitして、それで終わり。ところが、5人のチーム実習で初めて、Gitが本当は何のための道具なのかを思い知った。

01 — The red letters赤い文字に焦った

自分の担当画面を作り終えて、mainに取り込もうとした。git merge。すると、見たことのない英語が並んだ。

bash$ git merge feature/list
Auto-merging list.jsp
CONFLICT (content): Merge conflict in list.jsp
Automatic merge failed; fix conflicts and commit the result.

正直、最初は「自分が何か壊した」と思って血の気が引いた。でも、落ち着いてメッセージを読むと、Gitはただ「同じ行を2人が別々に直したから、どっちを残すか決めてくれ」と言っているだけだった。事故じゃなくて、相談だった。

02 — Reading the marks印を読む

ファイルを開くと、見慣れない記号が入っていた。これも、読めば意味は単純だった。

diff<<<<<<< HEAD
<h1>商品一覧</h1>        // 自分が書いた方
=======
<h1>商品リスト</h1>      // 相手が書いた方
>>>>>>> feature/list

=======の上が自分、下が相手。どちらを残すか——あるいは両方を踏まえて書き直すかを、人間が決める。記号を消して、正しい形に整えて、保存してコミットすれば終わり。やってみれば、こわくなかった。

コンフリクトはエラーじゃない。「ここは人が判断して」という、Gitからの引き継ぎだった。

03 — What it taught前職と重なったこと

これは、前職で複数人が同じ資料を触るときの「版管理」とまったく同じ問題だった。誰かが古いファイルを上書きして、修正が消える——あの事故を防ぐために、当時はファイル名に日付と担当を入れて運用していた。Gitは、それを仕組みとして、しかも履歴ごと安全にやってくれる。

チームで「先にpullしてからpushする」という約束を決めてからは、コンフリクト自体がほとんど起きなくなった。道具の前に、段取りの取り決めが効く。これも前職で何度も経験したことだった。

チームで決めたルール

① 作業前に必ず git pull。② 1機能 = 1ブランチ。③ こまめにコミットして、変更を小さく保つ。これだけでチーム開発がぐっと穏やかになった。

— 学習中に詰まったこと、解決した方法を、忘れないように書き留めています。間違いに気づいたら追記・修正していきます。
