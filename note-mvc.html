MVCがなかなか掴めなかった話

「Controllerに書くのか、Modelに書くのか」で毎回手が止まっていた。原因はコードじゃなくて、頭の中に流れの地図がなかったことだった。

サーブレットとJSPを習い始めて、最初の壁がMVCだった。教科書には「ModelとViewとControllerに分ける」と書いてある。言葉の意味は分かる。でも、いざ自分で課題を作ろうとすると、このコードはどこに書けばいいんだという手が、毎回止まった。

DBから一覧を取ってくる処理。これはModel。画面に出すのはJSP=View。じゃあ「一覧ページを開いたらDBを呼んで結果を渡す」というつなぎは？ ……ここがずっと曖昧だった。

01 — What I missedつまずいていた原因

結論から言うと、僕に足りていなかったのは知識じゃなくて「リクエストがどこを通っていくか」という地図だった。クラスの役割を単語で覚えても、処理の流れが見えていないと置き場所は決められない。

そこで、前職で資料の校正をしていたときの癖が役に立った。複雑な手順は、必ず一度フローにして紙に書き出す。今回も同じことをした。

この5ステップを声に出して読めるようになってから、「どこに書くか」で迷うことがほぼなくなった。Controllerは交通整理だけ。重い処理はModel。画面はView。 流れの中に置けば、役割は自然に決まる。

02 — In codeコードに落とすと

実際に書いた一覧表示のControllerは、驚くほど薄くなった。仕事をしていないように見えて、これでいい。

Java// Controller — 受け取って、振り分けて、渡すだけ
protected void doGet(HttpServletRequest req, HttpServletResponse res)
        throws ServletException, IOException {

    // Modelに任せる（Controllerは中身を知らない）
    ItemDao dao = new ItemDao();
    List<Item> items = dao.findAll();

    // Viewに渡して転送する
    req.setAttribute("items", items);
    req.getRequestDispatcher("/list.jsp")
       .forward(req, res);
}

ポイントは、ControllerがfindAll()の中身を一切知らないこと。SQLがどうなっているか、テーブルが何かは、Modelの責任。Controllerは「取ってきて」と頼んで「画面に渡す」だけ。

役割で迷ったら、コードを直す前に流れの図を見直す。たいてい、置き場所じゃなくて流れの理解がずれている。

03 — Why it clicked腑に落ちた理由

振り返ると、これは前職でやっていた業務の分担とまったく同じだった。受付の人が、自分で全部処理しようとせずに担当部署へ回す。資料を作る人と、それをチェックして発送する人を分ける。責務を分けると、どこで何が起きているか追える。バグが出ても「Modelか、Controllerか」で切り分けられる。

「分けると後で楽になる」という感覚は、8年間の事務仕事でさんざん体に入っていたものだった。技術用語が新しかっただけで、考え方は知っていた。そう気づいてから、学習がぐっと楽になった。

次にやること

いまは1つのServletに1つの処理。次は、URLごとの分岐をフロントコントローラ1枚にまとめる構成を試して、画面が増えてもController地獄にならない作りを学ぶ。
