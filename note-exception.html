例外を握りつぶさない

エラーが出ないように、とりあえずtry-catchで囲む。最初はそうしていた。でもそれは、問題を隠していただけだった。

Javaを書き始めたころ、赤いエラーが出ると怖くて、とにかくtry-catchで囲んでいた。エラーが画面から消えると安心した。でもある日、「動いているはずなのに結果がおかしい」課題で、まったく原因がつかめなくなった。

01 — Hiding the problem隠していただけだった

原因は、自分で書いたこのコードだった。

Javatry {
    int value = Integer.parseInt(input);
    // ... 処理 ...
} catch (Exception e) {
    // 何もしない ← これが地獄の入口だった
}

catchで受け取っておきながら、中で何もしていない。エラーは起きているのに、握りつぶされて、誰にも知らされない。表面上は「エラーが出ない」けれど、本当は「エラーが見えなくなっただけ」だった。これでは、どこで何が失敗したのか追いようがない。

エラーを消すことと、エラーを解決することは、まったく別物だった。

02 — Catch what you can handle対処できるものだけ受ける

学び直して、考え方を変えた。catchするのは「ここで対処できる」ときだけ。対処できないなら、握りつぶさずに、せめて記録するか、呼び出し元へ投げる。

Javatry {
    int value = Integer.parseInt(input);
} catch (NumberFormatException e) {
    // 想定できる失敗 → 利用者に分かる形で返す
    req.setAttribute("error", "数字を入力してください");
}

Exceptionでまとめて握るのをやめて、NumberFormatExceptionのように想定している失敗だけを名指しで受ける。そうすると、「ここでは入力ミスを想定している」という意図がコードから読み取れる。

03 — The proofreader's eye校正の目が効いた

これは、前職の校正の仕事と地続きだった。資料のチェックでいちばんやってはいけないのが、「たぶん大丈夫」で見逃すこと。怪しいところを握りつぶさず、必ず確認して、判断の根拠を残す。例外処理で「握りつぶさない」と知ったとき、ああ、これは知っている考え方だ、と思った。

いまの自分ルール

空のcatchは書かない。catchするなら「対処する」か「記録する」か「投げ直す」のどれか。エラーは、消すものじゃなくて、扱うもの。
